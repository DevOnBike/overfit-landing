Zero-allocation LLM inference in pure C#

Running GPT-2 and Llama-family models on .NET — no Python, no native binaries, zero bytes allocated per generated token. A build log.

The premise

For a lot of teams, "we'd like to use a language model" and "we need to run a Python service" are the same sentence. If your product is built in .NET, that second sentence is expensive: a Python runtime, a model server, usually a GPU, a separate deployment pipeline, a separate thing for security to audit — a whole parallel stack, bolted to your real one, to add a single feature.

I wanted to know how far you can get without it. Not "call a hosted API from C#" — actually run the model: in-process, in managed code, on the CPU you already have.

The result is Overfit — a deep-learning and inference engine written entirely in C#, on .NET 10. No native binaries. No Python. No ONNX Runtime dependency. It runs GPT-2 and the Qwen / Llama / Mistral families, and it fine-tunes them.

This is the build log of the part that was hardest, and the part I'm most proud of: getting transformer inference down to zero bytes allocated per generated token — and keeping it there, checked on every CI run.

Why "zero allocation" is the whole game

It sounds like a micro-optimization. It isn't.

.NET is a garbage-collected runtime. Every object you allocate is eventually collected, and a Gen-0 collection pauses your threads. In a batch job, nobody cares. In a service that runs inference inside a request, every allocation is a small deposit toward a GC pause — and that pause lands on some unlucky request as a tail-latency spike. You can't predict which one.

"0 bytes per operation" is not a vanity number. It's the property that turns unpredictable tail latency into predictable tail latency. If the hot path allocates nothing, Gen-0 never fills because of inference, and inference never causes a pause. That is the entire point — and it's a property you have to design for from the first line, because you cannot retrofit it.

Inference without a garbage truck

The basic move is caller-owned buffers. Overfit's inference entry point looks like this:

using var engine = InferenceEngine.FromSequential(model, inputSize: 784, outputSize: 10);

Span<float> input  = stackalloc float[784];
Span<float> output = stackalloc float[10];
engine.Run(input, output);   // zero allocation

The caller owns input and output. The engine never allocates them, never allocates scratch, never returns a freshly-minted array. Internally the math is Span<T>-based kernels over pooled storage; the working buffers are allocated once, when the engine is built, and reused forever after.

On a small model — Linear(784→10) — this comes out at 250 ns and 0 bytes per call, against 1,899 ns and 224 bytes for ONNX Runtime's pre-allocated path. The 7.6× is not because Overfit's matmul beats ONNX Runtime's — it absolutely does not. It's because at this size the per-call overhead dominates the actual math, and Overfit has none: no managed-to-native marshalling boundary, no per-call allocation. Under 8-way concurrency the honest headline is the allocation column: Overfit holds 0 bytes; ONNX Runtime allocates 117 MB of managed memory across the run.

The hard part: KV-cache decode at 0 B/token

A language model generates one token at a time, and each token attends to every token before it. The naive implementation re-runs the whole forward pass over the growing context for every new token — O(N²) work, and the allocations grow with the output. Overfit's own legacy path takes 6.3 seconds for 64 GPT-2 tokens and simply runs out of memory at 128.

The fix is the standard one — a KV-cache: keep each layer's computed keys and values, so each new token only computes its own. That's O(N). The hard part, given the zero-allocation rule, is where the memory lives.

Overfit's answer is a zero-copy weight strategy. When you create a session, the runtime does not copy the model's weights into a cache-friendly layout. It builds a tree of structs — StackWeights → BlockWeights → SingleHeadWeights — that hold ReadOnlySpan<float> references straight into the model's existing storage. Session creation allocates exactly one thing: the KV buffers, sized once for the full context (~74 MB for GPT-2 Small). After that, every single token decode allocates zero bytes. The weights are borrowed; the KV slots are pre-allocated; the logits buffer is caller-owned.

And because "0 B/token" is exactly the kind of claim that quietly rots, there's a test that measures per-token allocation and fails the build if anything regresses. The claim is executable, not aspirational.

The payoff: 64 tokens in 973 ms instead of 6,318 ms, and 128 tokens runs cleanly where the naive path is already dead.

The wall: Parallel.For allocates 3 KB per call

This is where it got interesting.

To use more than one core you reach for System.Threading.Tasks.Parallel.For. It's the obvious tool, and a good one — for a training loop. But Parallel.For allocates about 3 KB per call: the closure capturing your loop variables, the Task[] for the chunks, internal bookkeeping.

3 KB per call is invisible in a training step. It is fatal to a zero-allocation inference path. The first time I parallelized the LM-head projection — the big final matmul that turns hidden states into vocabulary logits — the 0-bytes-per-token number turned into ~92 KB across 31 tokens. Parallel.For and the zero-allocation contract are simply incompatible.

So inference parallelism needed a different dispatcher. I wrote one — OverfitParallelFor — and it has one genuinely interesting decision in it.

The structure is conventional: N worker threads spawned once at startup and parked; a function-pointer body (delegate*<int,int,void*,void>, so there's no closure to allocate); chunk descriptors the workers claim with an Interlocked increment. The interesting part is how you wake the workers.

The obvious way is to give each worker its own AutoResetEvent and Set() all of them. It works, and it's slow in a specific way: each Set() is a separate kernel transition, and they serialize. At 32-way fan-out you hit a hard floor around 32–47 µs just dispatching — before any of your actual work runs.

The fix is to stop signalling workers individually. All N workers wait on a single SemaphoreSlim. To wake them, you call SemaphoreSlim.Release(N) — one call that releases N permits. The OS scheduler then resumes the waiters roughly in parallel (futex_wake(N) on Linux, WakeConditionVariable on Windows) instead of one syscall at a time. Warm dispatch drops to about 5 µs, and — no closure, no Task[] — it allocates 0 bytes per call.

That single change — bulk-wake instead of per-worker signal — is what made allocation-free CPU parallelism actually work. With the dispatcher in place, the elementwise kernels (GELU, LayerNorm, attention) moved onto it, and a 4-layer GPT-1 training step at batch 32 dropped from 414 ms to 114 ms.

One honest caveat, because the result surprised me: it does not pay everywhere. I moved the residual Add onto the dispatcher and measured a regression. Add is memory-bandwidth-bound — two reads and a write — and 2–3 cores already saturate the memory bus, so the dispatch overhead is pure cost. Parallelism pays for compute-bound kernels, not bandwidth-bound ones. I kept the measurement and reverted the change.

Does it actually produce the right numbers?

It is easy to write a transformer that is fast and subtly wrong. A sign error, an off-by-one in the causal mask, a transposed weight — the model still generates something, and you don't notice for a long time.

So Overfit's GPT-2 import is checked against PyTorch. Same weights, same prompt; run the forward pass in both; compare the final-position logits. The result: top-10 logit overlap, 10 out of 10, and a maximum absolute difference of 0.000107 — the float32 noise floor. The two implementations agree as closely as two correct float32 implementations can. That comparison is an ordinary test; on a machine with the model fixture present, it runs on every dotnet test.

And the cheap sanity check — it generates coherent English:

The honest scorecard

What this approach wins: predictable, allocation-free inference; no native dependency and no Python runtime to deploy, secure, or audit; single-file Native-AOT deployment; it runs on ordinary CPUs and down to edge devices; and the data never leaves your process — there is no outbound call, by construction.

What it does not win: raw throughput on large-scale training. The same GPT-1 training step in PyTorch on CPU is still 2.2–3.6× faster — PyTorch's GEMM is Intel MKL, decades of hand-tuned AVX-512 assembly, and a pure-C# kernel written in a few months does not out-code that. Overfit's axis was never raw GEMM throughput. It is not GPU-first, and it is not for frontier-scale models. Pretending otherwise would only waste your time and mine.

What it's for, and where to get it

The honest "who": .NET teams that want to run AI without adopting a parallel Python stack; regulated and on-premises environments where the data legally cannot leave the building, so a hosted API is off the table; edge and embedded deployments where a 300 MB Python install isn't an option.

Concretely, today, Overfit runs GPT-2 and the Qwen / Llama / Mistral families — loading GGUF files straight from Ollama or HuggingFace, including Q4_K and Q6_K quantization — and it does LoRA fine-tuning, so you can adapt a model to your own data without any of it touching a third party.
